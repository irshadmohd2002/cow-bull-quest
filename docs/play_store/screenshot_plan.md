# Screenshot plan — six phone screenshots

Status: **completed plan only. No screenshots are generated by this
document** — capturing them is a separate, later action against an actual
release (or release-equivalent) build.

## General rules (apply to every screenshot)

- No emulator chrome (no device bezel, no emulator window controls) unless
  the chosen capture tool crops it out automatically.
- No debug banner ("DEBUG" ribbon) — capture from a release or profile
  build, never a plain debug build, per CLAUDE.md's build guidance.
- No email address, developer name, or other personal identifier visible
  anywhere on screen.
- No terminal windows, IDE overlays, or development tooling visible.
- No notification shade pulled down; no incoming notifications visible.
- No fake or aspirational features — every screen must show only what the
  shipped app actually does (no daily quests, leaderboards, coins,
  rewards, hints, or AI — see `branding_guide.md` "Prohibited uses").
- No misleading text (e.g. don't hand-edit or overlay claims onto a
  screenshot that aren't true of the running app).
- Screenshots must be captured from the actual release build (or a
  release-equivalent, e.g. profile build), reflecting real UI, not mockups
  — do not use `docs/design/cow_bull_quest_ui_mockup.png`, which contains
  unimplemented mockup-only features.
- Status bar: use a clean status bar (full battery or a typical charge
  level, full signal/Wi-Fi, standard time e.g. 9:41 or 12:00) — no low
  battery, no airplane mode, no active screen recording indicator.
- Crop/framing: capture at the device's native resolution; do not scale up
  a smaller screenshot. Keep the full screen (status bar through bottom
  edge) unless the chosen Play listing template requires cropping the
  status bar — if so, crop consistently across all six.

## 1. Home screen — word length and difficulty selection

- **State to prepare:** fresh app launch (or return-to-Home) with the
  Home screen's default selection visible — 4 letters, Common difficulty
  (`HomeScreen._defaultDifficulty`) — or deliberately select 5 letters and
  Hard to show the picker in an active, non-default state. Either is
  representative; pick one and keep it consistent with the caption.
- **Theme:** Light.
- **Suggested caption:** "Choose your word length and difficulty."
- **Sensitive-information check:** none — this screen shows no user data
  at all.
- **Status-bar considerations:** standard rules above.
- **Crop/framing guidance:** full screen, including the "Start Game" /
  "How to Play" / "Settings" / "Statistics" entry points below the fold if
  they fit; if not, prioritize the word-length and difficulty cards.
- **Must not appear:** any reference to features not in `HomeScreen`
  (e.g. no login button, no ad banner).

## 2. Active game — multiple accepted guesses

- **State to prepare:** start a game (5 letters, any difficulty
  recommended for a readable guess history) and submit at least 3-4 valid
  guesses so `GuessHistory` shows a populated list of bulls/cows results,
  without winning or losing yet (`GameActive` state, no validation
  banner showing).
- **Theme:** Light.
- **Suggested caption:** "Every guess earns Bulls and Cows clues."
- **Sensitive-information check:** the secret word is not revealed while
  a game is active (only guesses and their bull/cow counts are shown), so
  there is nothing sensitive on screen; use any set of valid dictionary
  guesses.
- **Status-bar considerations:** standard rules above.
- **Crop/framing guidance:** full screen; ensure the guess-history list
  and the guess input field are both visible.
- **Must not appear:** the validation-error banner (save that for
  screenshot 3); the completed-game outcome card.

## 3. Validation feedback after an invalid guess

- **State to prepare:** in an active game, submit one intentionally
  invalid guess (e.g. wrong length, or containing a non-letter) so the
  red `_ValidationBanner` is visible above the guess history.
- **Theme:** Light.
- **Suggested caption:** "Clear feedback when a guess isn't valid."
- **Sensitive-information check:** none; use a harmless invalid guess
  (e.g. too short/too long), not anything resembling personal data.
- **Status-bar considerations:** standard rules above.
- **Crop/framing guidance:** full screen, framed so the validation banner
  is clearly legible near the top of the guess area.
- **Must not appear:** any real error/crash dialog — this must be the
  app's own designed validation banner, not a Flutter error screen.

## 4. Winning completion screen

- **State to prepare:** play a game to a win (`GameCompleted` with
  `GameStatus.won`), showing the "You won!" card, the revealed secret
  word, and the attempts-used count.
- **Theme:** Dark or Light — pick Light for variety with screenshot 6
  covering Dark.
- **Suggested caption:** "Crack the code and see how you did."
- **Sensitive-information check:** the secret word shown here is a
  standard dictionary word chosen by the app itself for that play-through
  — not personal data — so this is safe to display; just don't caption it
  as if it were always the same word (results vary by playthrough).
- **Status-bar considerations:** standard rules above.
- **Crop/framing guidance:** full screen; make sure the "Restart" /
  "Return to Home" buttons are visible along with the outcome card.
- **Must not appear:** any losing-state UI in the same shot; any fake
  score/ranking element not actually in `_CompletedGameView`.

## 5. Statistics overview after several games

- **State to prepare:** play enough games (a mix of wins/losses, varied
  word lengths and difficulties) so the Statistics screen shows non-empty
  `StatisticsSummaryCard`, both breakdown sections ("By word length", "By
  difficulty"), and a populated `RecentGamesList`.
- **Theme:** Light.
- **Suggested caption:** "Track your wins, streaks, and recent games."
- **Sensitive-information check:** `CompletedGame` records contain no
  secret word or guess history — only word length, difficulty, outcome,
  and attempt counts — so this screen is safe to show in full; double
  check the recent-games list only shows those fields.
- **Status-bar considerations:** standard rules above.
- **Crop/framing guidance:** full screen; if content is long, scroll to
  frame the summary card and at least one breakdown section together
  rather than cutting the summary card in half.
- **Must not appear:** the "Clear statistics" confirmation dialog itself
  (that's a transient interaction, not a representative static screen).

## 6. Dark-theme Game or Rules screen

- **State to prepare:** switch theme preference to Dark in Settings, then
  either (a) return to an active game (same setup as screenshot 2) or (b)
  open the Rules ("How to Play") screen showing the Bulls/Cows scoring
  section and a worked example. Rules screen is recommended since it also
  demonstrates the app's own teaching content distinctly from screenshot 2.
- **Theme:** Dark.
- **Suggested caption:** "Play comfortably in light or dark mode."
- **Sensitive-information check:** none — Rules screen content is static,
  developer-authored explanatory text and worked examples (e.g. the
  built-in APPLE/AMPLE and SPEED/EERIE examples), not live gameplay data.
- **Status-bar considerations:** standard rules above; verify status-bar
  icon color/contrast looks correct against the dark background.
- **Crop/framing guidance:** full screen; if using Rules, frame the
  "Scoring" section (Bulls/Cows definitions) so it's legible at thumbnail
  size in the Play Store listing.
- **Must not appear:** any lingering light-theme element (verify the
  theme actually applied before capturing).
